# Задание 1. Исследование моделей и инфраструктуры для RAG-бота

## Контекст и целевая аудитория

**Кто даёт задачу:** руководство и архитектурная команда QuantumForge Software.  
**Для кого делается:** сотрудники компании — разработчики (архитектура, API, микросервисы), поддержка (типовые проблемы), менеджеры и аналитики (регламенты), новые сотрудники (онбординг).

**Ограничения кейса:** часть материалов конфиденциальна; компания готовится к SOC 2, данные клиентов (ABB, Wärtsilä, энергокомпании) чувствительны. Размещение в публичном облаке (OpenAI и т.п.) может быть неприемлемо для части документов — нужны варианты с локальной или гибридной обработкой.

## 1. Сравнение LLM-моделей

| Критерий | Локальные (Hugging Face: Llama, Mistral и др.) | Облачные OpenAI (актуальные модели) | YandexGPT |
|----------|-----------------------------------------------|-------------------------------------|-----------|
| **Качество ответов** | 7.5–8.1/10 по бенчмаркам; хуже следование инструкциям, больше галлюцинаций | Стабильно высокое качество и следование промптам | Среднее; для русского языка хорошая токенизация, по качеству уступает флагманским моделям OpenAI |
| **Скорость** | 1.8–2.5 с на ответ (зависит от железа и модели) | Обычно 1–2 с на ответ в зависимости от модели | Сопоставимо с облачными; асинхронный режим дешевле |
| **Стоимость владения** | Сервер (CPU/GPU), электричество, администрирование; нет платы за токены | Оплата за токены; при большом трафике — существенные расходы | От ~0.4 ₽/1K токенов; асинхронный режим примерно в 2 раза дешевле |
| **Развёртывание** | Сложно: нужны GPU, настройка инференса, обновления моделей | Минимально: API-ключ и интеграция | Просто: API, хорошая документация на русском |

**Вывод по LLM:** для корпоративного RAG с конфиденциальными данными и SOC 2 важны контроль над данными и возможность не отправлять их в публичное облако. Локальные модели (Mistral, Llama) дают конфиденциальность и предсказуемую стоимость при росте нагрузки; облачные (OpenAI/YandexGPT) — лучшее качество и простота старта. Рационально рассматривать гибрид: чувствительные запросы — локально, остальные — облако (в т.ч. YandexGPT для русскоязычного контента).

## 2. Сравнение моделей эмбеддингов

| Критерий | Sentence-Transformers (локально) | OpenAI Embeddings (облако) |
|----------|----------------------------------|----------------------------|
| **Скорость создания индекса** | Зависит от CPU/GPU на своей инфраструктуре; пакетная обработка под контролем | Быстро при наличии квот; ограничения API и сеть могут замедлять массовую индексацию |
| **Качество поиска** | Модели уровня all-MiniLM-L6-v2, all-roberta-large-v1 дают 83–85% на MTEB; мультиязычные модели подходят под смесь EN/RU | text-embedding-3-large: высокие баллы на MTEB/MIRACL; стабильно хорошее качество |
| **Стоимость** | Нет платы за токены; только инфраструктура | Около $0.13 за 1M токенов (large); 1M документов — порядка сотен долларов | 
| **Конфиденциальность** | Данные не покидают периметр | Тексты отправляются в OpenAI; неприемлемо для конфиденциальных регламентов и политик |

**Вывод по эмбеддингам:** для QuantumForge с учётом объёма (~18 000 MD + тысячи страниц Confluence + PDF) и требований к конфиденциальности предпочтительны **локальные Sentence-Transformers** (например, мультиязычная модель под EN/FI/RU). Это снижает стоимость при росте объёма и исключает утечку текстов в третьи стороны, что важно для SOC 2 и аудита.

*В рамках курса при реализации заданий 2–3 фактически использованы **облачные эмбеддинги Yandex Cloud**: модели `text-search-doc` (для индексации чанков) и `text-search-query` (для поисковых запросов), API [Foundation Models — textEmbedding](https://cloud.yandex.ru/docs/yandexgpt/api-ref/embeddings). Векторная БД — ChromaDB (персистентное хранилище на диске, метрика cosine), что соответствует выводу по векторным БД ниже.*

## 3. Сравнение векторных баз: ChromaDB и FAISS

### ChromaDB

- **Назначение:** полноценная векторная БД с хранилищем, персистентностью и метаданными.
- **Индексация:** HNSW (Hierarchical Navigable Small World); настраиваются batch size, sync threshold, метрики (L2, cosine, inner product).
- **Хранение:** по умолчанию SQLite; возможны бэкенды (в т.ч. DuckDB для разработки, ClickHouse для продакшена). У компании уже есть ClickHouse в стеке — возможна конвергенция.
- **Особенности:** фильтрация по метаданным (роль пользователя, тип документа, пространство Confluence), персистентность из коробки, простой API, мультимодальность.
- **Масштаб:** ориентир — до порядка 10M векторов на одном узле.
- **Скорость:** задержка запроса порядка 15 ms, пропускная способность порядка сотен QPS — достаточно для внутреннего корпоративного бота.

### FAISS

- **Назначение:** библиотека для быстрого поиска ближайших соседей, а не готовая БД.
- **Скорость:** 0.3 ms (CPU) / 0.05 ms (GPU), десятки тысяч QPS — существенно быстрее ChromaDB.
- **Масштаб:** миллиарды векторов при правильном индексе и GPU.
- **Недостатки для нашего кейса:** нет встроенной персистентности (нужно самостоятельно сохранять индексы и метаданные), слабая поддержка фильтрации по метаданным (роль, тип документа), выше сложность внедрения и сопровождения.
- **Стоимость владения:** при использовании GPU — дороже инфраструктура; при CPU — дешевле железа, но больше трудозатрат на свою обвязку (хранение, версионирование, бэкапы).

### Сводная таблица

| Критерий | ChromaDB | FAISS |
|----------|----------|--------|
| **Скорость поиска** | ~15 ms, сотни QPS | 0.05–0.3 ms, десятки тысяч QPS |
| **Скорость индексации** | Удобная пакетная загрузка, персистентность | Очень быстрая, но сохранение/загрузка — своя реализация |
| **Сложность внедрения и поддержки** | Низкая: готовый сервер/библиотека, документация | Выше: нужно строить слой персистентности, версионирования, бэкапов |
| **Удобство** | Коллекции, метаданные, фильтры из коробки | Нужно проектировать схему метаданных и фильтрацию самим |
| **Стоимость владения** | Обычно без GPU; можно на существующих EKS-нодах | Максимум скорости — с GPU; без GPU проще по деньгам, сложнее по коду |

**Вывод по векторным БД:** для корпоративного RAG-бота с персонализацией по ролям (разработчик, саппорт, менеджер, онбординг) и большим числом источников (Confluence, GitHub, PDF) важны **метаданные и фильтрация**. Требования к задержке — скорее «удобный ответ за 1–3 секунды», а не субмиллисекунды. Поэтому **предпочтительна ChromaDB**: быстрее выход в продакшен, меньше кастомного кода, естественная интеграция с конвейером RAG и возможность опоры на существующий ClickHouse при росте. FAISS имел бы смысл при десятках миллионов векторов и жёстких SLA по латентности.

## 4. Рекомендуемая конфигурация сервера для RAG-бота

Исходные данные: десятки тысяч документов, сотни одновременных пользователей (оценка), эмбеддинги на CPU или одной GPU, LLM — либо облачный API, либо локально (7B–13B с квантизацией).

**Вариант A — только эмбеддинги и векторная БД (LLM в облаке):**

- **CPU:** 8–16 ядер (например, Intel Xeon или аналог).
- **RAM:** 32–64 GB (индекс ChromaDB, модель Sentence-Transformers, кэш).
- **GPU:** не обязательна; при желании ускорить эмбеддинги — одна T4 или L4.
- **Диск:** NVMe, 500 GB–1 TB (индексы, кэш, логи).
- **Сеть:** стандартная для EKS; исходящий трафик к облачному LLM.

**Вариант B — полный локальный контур (эмбеддинги + локальная LLM):**

- **CPU:** 16+ ядер.
- **RAM:** 128–256 GB (особенно при 13B+ моделях).
- **GPU:** 1× L4 / A10 / T4 (24 GB) для инференса 7B–13B в 4-bit; для 8B в 4-bit достаточно 16 GB vGPU.
- **Диск:** NVMe 1 TB+.
- Подходит для сценариев, где запросы к LLM не должны уходить в облако.

**Вариант C — минимальный старт (пилот):**

- **CPU:** 4–8 ядер.
- **RAM:** 16–32 GB.
- **GPU:** не требуется.
- Эмбеддинги — облачный API (Yandex text-search-doc/query или OpenAI) или Sentence-Transformers на CPU; LLM — YandexGPT/OpenAI по API; векторная БД — ChromaDB на том же узле или в EKS.

**Рекомендация по шагу внедрения:** начать с варианта C на EKS, затем перейти к A или B в зависимости от решения по размещению LLM (облако vs on-prem) и объёма нагрузки.

## 5. Итоговые рекомендации: варианты конфигурации

### Вариант 1. «Облачный минимум» (быстрый старт) — *реализован в курсе*

- **LLM:** YandexGPT (русский язык, приемлемая цена) или OpenAI (актуальные модели).
- **Эмбеддинги:** Yandex Foundation Models (text-search-doc / text-search-query) или OpenAI Embeddings.
- **Векторная БД:** ChromaDB (персистентное хранилище на своей стороне).
- **Инфраструктура:** контейнеры в EKS или локально, скромные ресурсы (вариант C).
- **Плюсы:** быстрый запуск, мало своей инфраструктуры, фокус на архитектуре пайплайна RAG.  
- **Минусы:** конфиденциальные документы не должны уходить в облако; ограничения по SOC 2 и аудиту.

### Вариант 2. «Гибрид» (рекомендуемый для QuantumForge)

- **LLM:** чувствительные запросы — локальная модель (Mistral/Llama 7B–13B на GPU); остальные — YandexGPT/OpenAI.
- **Эмбеддинги:** Sentence-Transformers локально (мультиязычная модель).
- **Векторная БД:** **ChromaDB** с персистентным хранилищем (при росте — бэкенд ClickHouse).
- **Инфраструктура:** вариант A или B в EKS; при необходимости — выделенный пул для RAG.
- **Плюсы:** конфиденциальность там, где нужно; масштабируемость; единая точка входа для поиска по метаданным и ролям.  
- **Минусы:** сложнее, чем чисто облачный вариант; нужна настройка маршрутизации запросов (конфиденциально / нет).

### Вариант 3. «Полностью локальный»

- **LLM:** Mistral/Llama на GPU (вариант B).
- **Эмбеддинги:** Sentence-Transformers на том же кластере.
- **Векторная БД:** ChromaDB.
- **Плюсы:** максимум контроля и соответствия требованиям по данным и SOC 2.  
- **Минусы:** выше стоимость железа и администрирования; качество ответов может быть ниже облачных моделей.

### Вариант 4. «Максимальная производительность поиска»

- **LLM:** облако или локально — по политике.
- **Эмбеддинги:** локально.
- **Векторная БД:** FAISS (с собственной обвязкой персистентности и метаданных).
- **Плюсы:** максимальная скорость поиска и потенциал масштабирования до миллиардов векторов.  
- **Минусы:** большие трудозатраты на разработку и поддержку; избыточно для текущего объёма и требований по срокам выхода на продакшен.

## 6. Выбор векторной БД и итог

**Рекомендуемая векторная БД для RAG-бота QuantumForge Software — ChromaDB.**

**Обоснование:**

1. **Соответствие задаче:** нужны быстрые персонализированные ответы, выявление пробелов в документации и удобство для разных ролей. ChromaDB даёт фильтрацию по метаданным (роль, тип документа, источник) без дополнительной разработки.
2. **Объём и рост:** десятки тысяч документов и рост на сотни страниц в месяц укладываются в возможности ChromaDB (до порядка 10M векторов). При необходимости можно перейти на бэкенд ClickHouse, уже используемый в компании.
3. **Сроки и ресурсы:** цель — сократить время поиска и затраты; ChromaDB позволяет развернуть рабочий контур за недели, а не месяцы, в отличие от кастомной обвязки вокруг FAISS.
4. **Конфиденциальность и SOC 2:** ChromaDB разворачивается на своей инфраструктуре (EKS), данные не передаются третьим сторонам; это упрощает обоснование при аудитах.
5. **Стек компании:** опыт с AWS, EKS, ClickHouse и монорепой на GitHub хорошо стыкуется с развёртыванием ChromaDB в Kubernetes и возможной миграцией на ClickHouse как бэкенд.

FAISS в данной постановке не выбирается, так как выигрыш в латентности поиска не критичен для внутреннего бота, а затраты на реализацию персистентности, метаданных и фильтрации по ролям существенно выше, чем выгода.

Итоговая рекомендуемая конфигурация для первого этапа: **вариант 2 (гибрид)** с ChromaDB в качестве векторной БД, локальными Sentence-Transformers для эмбеддингов и разделением LLM (локально для конфиденциальных запросов, YandexGPT/OpenAI — для остальных) с последующей оценкой качества и затрат и при необходимости сдвигом в сторону варианта 3 (полностью локальный).

**Реализация в рамках курса.** Рекомендации выше ориентированы на производственный кейс QuantumForge, где приоритет — конфиденциальность и соответствие SOC 2; для него обоснован гибридный или полностью локальный контур. В учебных заданиях курса сделан осознанный выбор в пользу **облачных API для LLM и эмбеддингов** (YandexGPT, Yandex Foundation Models text-search-doc/query): это смещает фокус на проектирование пайплайна RAG, интеграцию векторного поиска и приложения без затрат времени на развёртывание и отладку локальных моделей. Векторная БД (ChromaDB) и приложение остаются на своей стороне — тем самым сохраняется граница «своя инфраструктура vs облачные сервисы» и переносимость выводов отчёта на реальные сценарии, где при необходимости те же компоненты можно заменить на локальные эмбеддинги и LLM в соответствии с вариантами 2 или 3.
